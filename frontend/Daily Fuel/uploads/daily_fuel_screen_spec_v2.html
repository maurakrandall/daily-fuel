My Go-Tos
Grab-and-go staples
UI elements
  • Header: "My Go-Tos"
  • Subtext: "These are your grab-and-go staples — the foods you reach for without thinking. Tap the ones you already eat."
  • Pill buttons — multi-select, tap to toggle
  • 8 items: Protein shake, Protein bar, Protein powder, Hard-boiled eggs, Mixed nuts, Beef/turkey jerky, Bone broth, String cheese
  • Selected state: filled/highlighted (dark navy)
  • Unselected state: outlined/light
  • "Next" button (always enabled — zero selections is valid)
Logic + notes
Go-Tos are prioritized in Rough Day plans and appear first in all plan outputs.

No minimum selection — user can skip entirely. If no Go-Tos selected, Rough Day plans draw from Will Eat pool with lowest effort items.

Unselected items are NOT Go-Tos. They do not carry over to Will Eat — they're simply not selected as grab-and-go staples. The user may still select them in the Will Eat screen.

Will Eat / Won't Eat
Full food profile — one screen
UI elements
  • Header: "Will Eat / Won't Eat"
  • Subtext: "Tap the foods that work for you. Skip the ones that don't."
  • All foods displayed as tappable pills organized by sub-group:
  • — Protein-primary (13 items)
  • — High-value protein+fiber (5 items)
  • — Nuts & Seeds (12 items)
  • — Vegetables & Fruit (13 items)
  • — Support foods (3 items)

Copy direction
Keep it light. Nothing is permanent — they can always change it later. Sub-group labels are uppercase, small, organizational only (PROTEIN-PRIMARY, NUTS & SEEDS, etc.).

"Won't eat" is never permanent — this is a product principle. GLP-1 preferences shift over time. Editability is built in via My Foods.
Logic
Will Eat eligible for daily plan recommendations

Won't Eat (untapped) filtered from all plans. Never surfaced in suggestions.

State persistence These selections are the single source of truth for plan generation, My Foods display, and Close the Gap filtering. Must persist correctly across all screens.

QA finding (May 19): Onboarding selections must properly persist to plan and My Foods screens. Bug identified where unselected items appeared in plans.

UI — food cards + interaction
Food items displayed as suggestion cards. No checkboxes. Each card shows:
  • Food name
  • Benefit label ("Easy protein, zero prep" / "Efficient protein & fiber" / "Quick energy stabilizer")
  • Role tag (PROTEIN / FIBER / PROTEIN + FIBER)

Tap interaction: User taps a card →
1. Micro-celebration animation (warm gold glow/pop)
2. Per-item values appear: "~9g protein + ~4g fiber!"
3. Card transitions to selected state (highlighted, subtle checkmark)
4. Running estimated total updates at top
5. Bar fills further

Tap again to deselect. Total adjusts down. No negative language.

All numbers use ~ or "about." USDA approximate values per standard serving. Never precise.
Logic — plan generation
V1 — rule-based

Inputs: daily state + food profile + targets

Selection priority
1. Protein+fiber doubles first (most efficient)
2. Fill protein gap from protein-primary items
3. Fill fiber gap from fiber-primary items

Rough day Go-Tos first, lowest effort, 3–4 items max, support foods eligible

Not hungry Easy/no-prep, lean toward doubles, 4–5 items

Feeling good Full Will Eat pool, more variety, 5–6 items

Critical filter ONLY foods from user's Go-Tos + Will Eat selections. Won't Eat items NEVER appear.

V2 AI-driven plan generation. Architecture session June 5.
Schema: Each food item needs: name, category (go-to / will-eat / supplement), role (protein / fiber / both / supplement / none), protein_g per serving, fiber_g per serving, effort_level (grab-and-go / minimal-prep / needs-cooking), serving_size_label.

The user's home screen once a plan exists
UI elements
This is what the user sees when they return to the app after saving a plan. It's also what they see after the celebration screen.

  • Header: "Today's Plan"

  • Running estimated total: "~Xg protein · ~Xg fiber"

  • Supplements reminder
  • "Close the gap" button — always available
  • Navigation to My Foods and My Goals
Persistence logic
Home screen logic
App open → check today's date → plan exists? → show saved plan view.
No plan? → show state selector.

No "Done for today" The user simply closes the app. Their plan persists. They can return anytime to see it or add more via Close the Gap. Plan resets automatically at midnight (date-based).

UX restructure (May 19): Removed "Done for today" entirely. The plan is a persistent daily artifact, not a one-shot interaction. The user owns their plan all day.
Minimal session (no foods)
If the user selected a state but tapped "Not feeling it right now" (no foods), the saved plan view shows:

— Their state selection
— "No foods selected yet"
— "Add foods" button → returns to the plan suggestion screen (B) with their original state

This means the user can come back at 2pm when they feel better and pick up where they left off, without re-selecting their state.

Flow + logic
Flow
"Save to my plan" → celebration screen (C) with COMBINED totals from all rounds → "See my plan" → saved plan view (C2) with everything added. "Close the gap" remains available on the saved plan for another round if more foods exist.

QA finding (May 19): Previous implementation created a redundant "Done for today" loop. Fixed: persistent plan model eliminates the loop entirely. User saves, celebrates, returns to their plan.

Product principle: the first round is personal (user's foods). The second round is discovery (universal suggestions). The bridge to My Foods means every Close the Gap session can permanently improve future plans.

Non-goals — never in any version
Product non-goals
  • Calorie data, calorie targets, calories burned — never, full stop
  • Comorbidity tracking (diabetes, A1C) — medical device territory
  • Specific medical advice or dosing guidance
  • Blood glucose tracking
  • Streaks, scores, or red numbers
  • Deficit language ("remaining," "left to go," "behind")
  • Warning colors (red, orange) anywhere in the UI
Product principles protecting these
"Daily Fuel is not inventing the anxiety. It is reducing it."

Two products exist inside Daily Fuel: Product A (hard-day companion) and Product B (community platform). Every decision protects Product A.

The user sees what they're gaining, not what they're missing. No target displayed. No deficit. Just a number that grows and feels good. That's celebration, not compliance.
